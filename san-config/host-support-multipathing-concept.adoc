= Compatibilidad de ONTAP con el acceso multivía del host SAN
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
ONTAP utiliza software ALUA (Acceso Asimétrico de Unidad Lógica) para multivía con hosts FC e iSCSI.

A partir de ONTAP 9,5, la conmutación/devolución de la pareja de alta disponibilidad multivía es compatible con los hosts NVMe que usan Asynchronous Namespace Access (ANA). En ONTAP 9,4, NVMe solo admite una ruta desde el host al destino, por lo que el host de aplicaciones debe gestionar la ruta al respaldo de su partner de alta disponibilidad.

Para obtener información sobre las configuraciones de host específicas compatibles con ALUA o ANA, consulte el https://mysupport.netapp.com/matrix["Herramienta de matriz de interoperabilidad de NetApp"^] y https://docs.netapp.com/us-en/ontap-sanhost/index.html["Configuración de host SAN ONTAP"] del sistema operativo del host.

El software de multivía, también conocido como software MPIO (I/O multivía), se recomienda en configuraciones de HA. El software de multivía es necesario en tu host SAN para que pueda acceder a un LUN o espacio de nombres NVMe a través de más de una ruta. Presenta un único disco al sistema operativo para todas las rutas a un LUN o espacio de nombres NVMe. Sin él, el sistema operativo podría tratar cada ruta como un disco independiente, lo que provocaría corrupción de datos.

Se considera que su solución tiene varias rutas si dispone de alguna de las siguientes opciones:

* Un único puerto iniciador del host conectando varios LIF SAN en la SVM
* Varios puertos de iniciador conectando a un único LIF SAN en la SVM
* Varios puertos de iniciador conectando varios LIF SAN en la SVM


Deberías usar Selective LUN Map, la división en zonas de conmutadores FC o conjuntos de puertos para limitar las rutas utilizadas para acceder a los LUN y a los espacios de nombres.

== Número recomendado de rutas desde el host a los nodos en el clúster

No debe superar más de ocho rutas desde el host a cada nodo del clúster. Tampoco debe superar la cantidad total de rutas que se pueden admitir para el sistema operativo del host y la multivía utilizada en el host.

Si tiene cuatro o más nodos en el clúster o más de cuatro puertos de destino que utilizan las SVM en cualquiera de los nodos, Puede usar los siguientes métodos para limitar el número de rutas que se pueden utilizar para acceder a las LUN de los nodos de modo que no supere el máximo recomendado de ocho rutas.

* link:selective-lun-map-concept.html["Asignación de LUN selectiva (SLM)"]
+
SLM reduce el número de rutas desde el host al LUN a solo las rutas en el nodo propietario del LUN y el nodo de alta disponibilidad (HA) asociado. SLM está habilitado por defecto. Deberías tener un mínimo de dos rutas por LUN que conecten a cada nodo de informes a través de SLM que está siendo usado por la máquina virtual de almacenamiento (SVM) en tu clúster. Esto elimina los puntos únicos de fallo y permite que el sistema sobreviva a fallos de componentes.

* link:../san-admin/limit-lun-access-portsets-igroups-concept.html["Conjuntos de puertos"]
+
Los conjuntos de puertos se pueden utilizar con SLM para restringir aún más el acceso de ciertos destinos a ciertos iniciadores. Cuando se utiliza SLM con conjuntos de puertos, se podrá acceder a los LUN en el conjunto de puertos del nodo que posee la LUN y en el partner de alta disponibilidad de ese nodo.

* Asignaciones de grupos de iniciadores (igroup) de tu host
+
Puedes definir igroups y asignarlos a LUN para controlar qué iniciadores tienen acceso a los LUN.

* link:../san-config/fc-fcoe-recommended-zoning-configuration.html["División en zonas de switches FC"]
+
Deberías crear una configuración de zonas si tu host no tiene instalada una solución de multivía, si hay cuatro o más hosts conectados a tu SAN o si SLM no está implementado en los nodos de tu clúster.
