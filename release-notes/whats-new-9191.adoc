== Redes

[cols="30%,70%"]
|===
| Actualizar | Descripción 


 a| 
Gestiona las versiones de NFS que admite el SVM
 a| 
La compatibilidad con NFSv4.2 se gestiona independientemente de la configuración de NFSv4.1, lo que permite a cada SVM anunciar o suprimir las capacidades de NFSv4.2 según sea necesario. Por defecto, NFSv4.2 está activado cuando se crea un nuevo servicio de NFS, a menos que el administrador lo desactive explícitamente.

* https://docs.netapp.com/us-en/ontap/nfs-admin/enable-disable-nfsv42-task.html["Activar o desactivar NFSv4.2 para los SVM de ONTAP"]
* https://docs.netapp.com/us-en/ontap/task_nas_enable_linux_nfs.html["Habilite el almacenamiento NAS para servidores Linux mediante NFS con ONTAP System Manager"]




 a| 
xref:../concepts/path-failover-concept.html#nas-path-failover[Mejora de la conmutación por error de LIF]
 a| 
La política de grupo de conmutación por error definida por el sistema de ONTAP amplía dinámicamente los objetivos de conmutación por error de los LIF de datos NAS para incluir potencialmente cualquier puerto disponible en el dominio de difusión de cualquier nodo disponible en el clúster. Puedes aplicar la política definida por el sistema a cualquier grupo de conmutación por error. Antes de ONTAP 9.19.1, los LIF con esta política solo podían conmutar por error a otro nodo en el clúster; si ese nodo se quedaba sin quórum, el LIF no conmutaba por error, aunque hubiera otros nodos disponibles.

== SAN

[cols="30%,70%"]
|===
| Actualizar | Descripción 


 a| 
xref:../san-config/configure-fc-nvme-hosts-ha-pairs-reference.html[Soporte para configuraciones FC de conexión directa]
 a| 
Los switches FC no son necesarios para conectar hosts FC y FC-NVMe al almacenamiento ONTAP. Los hosts pueden conectarse directamente a los puertos adaptadores FC en los sistemas AFF, ASA y FAS. Esta conexión directa es especialmente útil en sitios remotos y entornos híbridos donde la simplicidad es importante.



 a| 
xref:../san-admin/multipath-automatic-lif-failover-support.html[Soporte para multivía activa-activa en sistemas AFF]
 a| 
Los sistemas AFF admiten multivía activo-activo en entornos SAN. En las configuraciones activo-activo, todas las rutas dentro del par HA que posee el LUN o el espacio de nombres están optimizadas para activo, de modo que en caso de conmutación por error de almacenamiento, el host no necesita esperar a la transición ALUA o ANA de las rutas de conmutación por error para reanudar la E/S. Esto reduce el tiempo hasta la conmutación por error, lo que permite a los clientes de SAN ejecutar cargas de trabajo de misión crítica en sistemas AFF con la misma resiliencia de rutas y equilibrio de carga sin interrupciones que antes solo estaba disponible en sistemas ASA.



 a| 
xref:../san-admin/multipath-automatic-lif-failover-support.html[Compatibilidad mejorada con la conmutación por error automática de SAN LIF]
 a| 
En los sistemas AFF y ASA se admite la conmutación por error automática de LIF para LIF de protocolo iSCSI en configuraciones de multivía activa-activa y activa-activa-scsi-only y para LIF de protocolo NVMe/TCP en configuraciones de multivía activa-activa.



 a| 
link:https://docs.netapp.com/us-en/ontap-fli/get-started/learn-about-data-migration.html["Compatibilidad con la importación de LUN extranjeros para el protocolo iSCSI"^]
 a| 
Foreign LUN Import (FLI) es compatible con el protocolo iSCSI para la conectividad backend.

== Seguridad

[cols="30%,70%"]
|===
| Actualizar | Descripción 


 a| 
Características de la carga de trabajo ARP eliminadas de la interfaz de usuario de System Manager
 a| 
System Manager ya no muestra las características de la carga de trabajo de Autonomous Ransomware Protection (ARP), incluidas las estadísticas de línea de base y de sobrecarga. Esta información se ha eliminado porque la actividad de sobrecarga se interpretaba erróneamente con frecuencia como un ataque real de ransomware o como algo que requería un ajuste manual.

El comportamiento y la detección de ARP no cambian, y solo se ve afectada la presentación de estos detalles por parte de System Manager. Las estadísticas Surge siguen estando disponibles mediante las opciones CLI y API de REST de ONTAP para análisis avanzados



 a| 
xref:../anti-ransomware/use-cases-restrictions-concept.html#supported-configurations[Protección autónoma contra ransomware compatible con SnapMirror synchronous y SnapMirror active sync]
 a| 
ARP/AI es compatible con volúmenes que participan en SnapMirror síncrono (NAS y SAN) y en relaciones SAN de sincronización activa de SnapMirror en plataformas compatibles. ARP/AI puede proteger volúmenes primarios en relaciones SnapMirror síncronas en sistemas FAS, AFF y AFX, y volúmenes primarios en relaciones SAN de sincronización activa de SnapMirror en sistemas AFF y ASA r2. ARP/AI se ejecuta en el volumen primario activo de lectura y escritura de la relación, creando instantáneas ARP solo primarias y proporcionando detección de ransomware basada en aprendizaje automático.



 a| 
xref:../networking/configure-tls-offload.html[Configuración de la descarga TLS]
 a| 
Puedes configurar la descarga de TLS para mejorar el rendimiento de TLS post-handshake aprovechando los recursos de las tarjetas Ethernet compatibles. Esta función descarga el cifrado y descifrado, reduciendo la sobrecarga de la CPU y mejorando el rendimiento.

== Eficiencia del almacenamiento

[cols="30%,70%"]
|===
| Actualizar | Descripción 


 a| 
xref:../flexcache/index.html[Mejora del rendimiento del NAS]
 a| 
ONTAP 9.19.1 permite más recursos de CPU para procesar solicitudes de listado de directorios NAS pequeños. Esta mejora beneficia a los clientes con cientos o miles de clientes que listan simultáneamente el mismo directorio NAS pequeño, con un tamaño de 2 MB o menos (aproximadamente 25,000 archivos).

Esta mejora reduce la necesidad de utilizar soluciones alternativas, como los volúmenes locales de FlexCache, para admitir un elevado número de clientes que realicen listados de directorios. En el caso de un menor número de clientes, el rendimiento no se verá afectado negativamente. Esta mejora del rendimiento es automática y no requiere configuración.

|===
